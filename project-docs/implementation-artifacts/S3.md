## Acceptance Criteria

### AC1: Single node failure with retry
```
Given a DAG run is in progress
And a single node fails on first attempt
When the retry mechanism triggers
Then it retries the same node (up to maxRetries)
And the run continues without aborting
```

### AC2: Multi-node failure exhausting retries
```
Given a DAG run is in progress
And a node fails repeatedly (exhausting all retries)
When retries are exhausted
Then the node is marked as failed
And the level continues with remaining nodes
And the failure is recorded in the state doc
```

### AC3: Critical path node failure → fallback agent
```
Given a DAG run is in progress
And a critical path node fails after retries
When the fallback agent is available
Then the node is resubmitted to the fallback agent
And the critical path is recalculated
```

### AC4: Level-wide failure → graph abort
```
Given a DAG run is in progress
And all nodes in a level fail with no recovery
When the last node fails
Then the graph is aborted
And the state doc shows ABORTED status
And the user is notified with failure details
```

### AC5: Graceful handling of edge cases
```
Given edge case conditions
When the DAG scheduler encounters:
  - Empty story list
  - Single-node graph
  - Self-referencing dependency
  - Missing story file
  - tmux spawn failure
Then it handles each gracefully without crashing
And produces a meaningful error message or status
```

## Implementation Plan

2. **Failure propagation** — verify state doc records failed nodes correctly
3. **DAG-aware recovery** — check downstream blocking logic in orchestrator
4. **Edge case inputs** — empty graphs, single nodes, self-deps
